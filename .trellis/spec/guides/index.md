# 思维指南

> **目的**：扩展你的思维，帮助你发现可能忽略的问题。

---

## 为什么需要思维指南？

**大多数 bug 和技术债务源于"没想到"**，而非技能不足：

- 没想到层级边界处会发生什么 → 跨层 bug
- 没想到代码模式会重复 → 到处重复的代码
- 没想到边界情况 → 运行时错误
- 没想到未来的维护者 → 难以阅读的代码

这些指南帮助你在**编码前提出正确的问题**。

## 快速参考：思维触发条件

### 何时考虑跨层问题

- [ ] 功能涉及 3 个以上层级（API、Service、Component、Database）
- [ ] 数据格式在层级间发生变化
- [ ] 多个消费者需要相同的数据
- [ ] 你不确定应该把某些逻辑放在哪里

→ 阅读 [跨层思维指南](./cross-layer-thinking-guide.md)

### 何时考虑代码复用

- [ ] 你正在编写与现有代码相似的代码
- [ ] 你看到相同的模式重复了 3 次以上
- [ ] 你正在向多个地方添加同一个新字段
- [ ] **你正在修改任何常量或配置**
- [ ] **你正在创建新的工具/辅助函数** ← 先搜索！

→ 阅读 [代码复用思维指南](./code-reuse-thinking-guide.md)

## 修改前规则（关键）

> **在修改任何值之前，务必先搜索！**

```bash
# 搜索你将要修改的值
grep -r "value_to_change" .
```

这个简单的习惯可以防止大多数"忘记更新某处"的 bug。

---

## 如何使用此目录

1. **编码前**：浏览相关的思维指南
2. **编码中**：如果感觉某个地方重复或复杂，查阅指南
3. **修复 bug 后**：将新的见解添加到相关指南中（从错误中学习）
