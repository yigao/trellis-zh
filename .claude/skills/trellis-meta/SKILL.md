# Trellis Meta

本 skill 面向已在项目中运行过 `trellis init` 的本地 Trellis 用户。AI 在阅读后应理解 Trellis 架构、运行模型以及该用户项目中的定制入口点，然后根据用户请求修改生成的 `.trellis/` 和平台目录文件。

默认操作范围为用户项目中的本地文件：

- `.trellis/`：workflow、config、task、spec、workspace（工作区）、scripts 以及运行时状态。
- 平台目录：`.claude/`、`.codex/`、`.cursor/`、`.opencode/`、`.kiro/`、`.gemini/`、`.qoder/`、`.codebuddy/`、`.github/`、`.factory/`、`.pi/`、`.kilocode/`、`.agent/`、`.windsurf/` 及类似目录。
- 共享 skill 层：`.agents/skills/`。

不要假设用户拥有 Trellis 源码仓库（repository）。不要默认修改全局 npm 安装目录或 `node_modules`。

## 如何使用

1. 首先阅读 `references/local-architecture/overview.md` 以建立本地 Trellis 系统模型。
2. 如果请求涉及特定 AI 工具，阅读 `references/platform-files/platform-map.md` 以及相关平台文件说明。
3. 如果用户想更改行为，阅读 `references/customize-local/overview.md`，然后打开具体的定制主题。
4. 在编辑之前，读取用户项目中的实际文件，并将本地内容视为权威来源。

## 当前规则

- `.trellis/workflow.md` 是本地 workflow 的单一事实来源。
- `.trellis/config.yaml` 是项目级 Trellis 配置和 task hook 配置入口点。
- `.trellis/spec/` 存储用户项目特定的编码 convention 和设计约束。
- `.trellis/tasks/` 存储 task PRD、技术说明、research 文件和 JSONL context。
- `.trellis/workspace/` 存储 developer 的 journal 和跨 session 记忆。
- 平台 settings/config 文件决定哪些 hook、agent、skill、命令、prompt 和 workflow 实际运行。
- `.trellis/.template-hashes.json` 和 `.trellis/.runtime/` 是管理/运行时状态文件。编辑前请确认必要性。

## 不要做的事

- 不要将 Trellis 上游源码视为本地定制的默认目标。
- 不要为实现项目需求而修改全局 npm 安装目录或 `node_modules/@mindfoldhq/trellis`。
- 不要用默认模板覆盖用户已修改的本地文件。
- 不要将团队私有项目规则放入公共的 `trellis-meta`；将项目规则放入 `.trellis/spec/` 或项目本地 skill。
- 不要将已废弃的历史机制描述为当前 Trellis 行为。
