# 本地工作流系统

`.trellis/workflow.md` 是用户项目内的 Trellis 工作流事实来源。AI 无需 Trellis 源代码即可理解当前项目应如何推进任务；有此文件足矣。

## 文件的职责

`.trellis/workflow.md` 有三项职责：

1. **说明工作流阶段**：规划、执行、完成。
2. **定义技能路由**：当用户表达特定意图时，AI 应使用哪个技能或智能体。
3. **提供 workflow-state 提示块**：钩子可以将当前状态的提示块注入到对话中。

## 当前阶段模型

```text
Phase 1: Plan    -> 明确要构建什么，产出 prd.md 和必要的研究
Phase 2: Execute -> 对照 PRD 和规范进行实现，然后检查
Phase 3: Finish  -> 最终验证，保留经验教训，完成收尾
```

每个阶段包含带编号的步骤，例如 `1.3 Configure context`。这些编号不是 `task.json` 中的运行时字段；它们是供 AI 和人类阅读的工作流结构。

## 技能路由

`workflow.md` 按平台能力区分路由：

- 支持子智能体的平台：默认派发 `trellis-implement` 进行实现，派发 `trellis-check` 进行检查。
- 不支持子智能体的平台：主会话读取 `trellis-before-dev` 等技能，然后直接执行。

更改本地 AI 行为时，首先更新 `workflow.md` 中的路由描述，然后检查对应的平台技能、命令或智能体文件是否需要保持同步。

## Workflow-State 提示块

`workflow.md` 底部可以包含如下状态块：

```text
[workflow-state:no_task]
...
[/workflow-state:no_task]
```

钩子根据当前任务状态选择正确的块并将其注入到对话中。常见状态包括：

| 状态 | 含义 |
| --- | --- |
| `no_task` | 当前会话没有活动任务。 |
| `planning` | 任务仍处于需求、研究或上下文配置阶段。 |
| `in_progress` | 任务已进入实现和检查阶段。 |
| `completed` | 任务已完成，等待收尾或归档。 |

如果用户想要更改策略，例如"没有任务时是否创建任务"、"何时可以跳过任务创建"或"是否必须使用子智能体"，编辑这些状态块及其上方的路由表。

## 本地修改模式

常见变更：

| 目标 | 编辑点 |
| --- | --- |
| 添加一个阶段 | 更新阶段索引、阶段正文、路由和状态块。 |
| 更改任务创建策略 | 更新 `no_task` 状态块和阶段 1 的描述。 |
| 更改默认的实现/检查路径 | 更新阶段 2 和技能路由。 |
| 更改收尾流程 | 更新阶段 3 和与 finish-work 相关的描述。注意当前划分：阶段 3.4 = AI 驱动的代码提交（批量、需用户确认），阶段 3.5 = `/finish-work`（归档 + 记录会话）。工作树不干净时 `/finish-work` 拒绝运行。 |
| 更改平台差异 | 更新按平台分组的路由描述。 |

编辑后，让 AI 重新读取 `.trellis/workflow.md`；不要假设旧对话中的流程仍然有效。

## 与平台文件的关系

`workflow.md` 是本地工作流的语义中心，但每个平台也可以有各自的入口文件：

- 技能，如 `trellis-brainstorm` 和 `trellis-check`。
- 命令/提示/工作流，如 continue 和 finish-work。
- 钩子，如 session-start 或 workflow-state 注入。

如果只修改了 `workflow.md`，平台入口文件可能仍包含旧的语言表述。当用户想要更改"AI 实际做什么"时，也要检查相关的平台目录。
